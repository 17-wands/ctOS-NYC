# Overlord Design System

A cold, technical, command-oriented design system for organizations that need to feel omnipresent, precise, and operationally inevitable. Overlord is built for surveillance-adjacent software, critical infrastructure platforms, industrial control rooms, security operations, municipal systems, command centers, and AI-enabled decision systems where authority, order, and machine intelligence define the experience.

Overlord should feel like a system that has already won: quiet, total, legible, and impossible to ignore.

## 1. Brand Positioning

### Core idea
Total visibility creates total control.

Overlord presents technology as an administrative layer over complex environments. It does not perform excitement. It does not ask for emotional buy-in. It establishes confidence through structured information, restrained typography, high-contrast UI, and a sense that every actor, object, location, and event can be classified.

The brand should feel institutional, technical, and nearly authoritarian without becoming visually crude. Its authority comes from restraint, precision, and systemic awareness.

### Brand promise
Observe everything. Classify instantly. Act with certainty.

### Audience
- Security operators, infrastructure leaders, government buyers, defense and public-safety teams, compliance teams, and enterprise risk leaders.
- Technical stakeholders responsible for complex environments where latency, ambiguity, and missed signals create risk.
- Decision-makers who need proof of control rather than emotional persuasion.

### Personality traits
- Cold
- Exacting
- Omniscient
- Procedural
- Institutional
- Severe
- Unsentimental
- Commanding

### What the brand should avoid
- Warm lifestyle imagery
- Friendly mascot systems
- Playful illustration
- Startup-style optimism
- Excessive gradients
- Cyberpunk clichés without operational purpose
- “Magic AI” messaging
- Hand-drawn or organic marks unless used as threat annotations
- Overly human language in critical UI

## 2. Voice and Messaging

### Voice principles
1. **Authority over enthusiasm** — the system speaks in declarations, not encouragement.
2. **Classification over description** — everything is categorized, ranked, scored, and assigned status.
3. **Evidence over persuasion** — use timestamps, telemetry, confidence, source, jurisdiction, and severity.
4. **Operational brevity** — reduce sentence length under pressure.
5. **Controlled menace** — the brand may feel intimidating, but it should never sound theatrical.

### Message architecture

#### Hero formula
`Operational control for environments that cannot go dark.`

Examples:
- Real-time command for critical infrastructure.
- Continuous visibility across people, assets, and events.
- The operating layer for high-risk environments.

#### Subhead formula
`Unify sensor data, identity, location, and response workflows into one authoritative command layer.`

Examples:
- Convert fragmented telemetry into a single live operating picture.
- Detect anomalies, classify risk, and coordinate response from one controlled interface.
- Bring infrastructure, security, and intelligence signals under one system of record.

#### Proof points
- Persistent monitoring
- Real-time anomaly detection
- Asset and identity correlation
- Automated escalation
- Geo-temporal event tracking
- Controlled access and audit trails
- Command workflows
- Sensor fusion
- Risk scoring
- Response orchestration

### Tone by surface
- **Homepage:** severe, confident, high-level, institutionally credible.
- **Product pages:** system-first, technical, operational, proof-led.
- **Sales decks:** controlled, strategic, outcome-oriented.
- **Docs:** direct, procedural, unambiguous.
- **Dashboards:** minimal, status-oriented, terse.
- **Alerts:** clipped, urgent, structured.
- **Errors:** factual, diagnostic, never apologetic.

### Preferred words
Observe, classify, monitor, detect, authorize, restrict, verify, correlate, resolve, command, control, status, signal, asset, node, sector, perimeter, subject, event, protocol, telemetry, anomaly, escalation, compliance, audit, incident, jurisdiction.

### Avoided words
Fun, delightful, magical, playful, friendly, community, vibes, explore, casual, whimsical, cute, journey, empower, unleash, amazing, revolutionary, frictionless.

## 3. Visual Direction

### Visual thesis
Overlord should look like administrative power translated into interface design. The system is not decorative. It is functional, extractive, and always watching.

The visual language should combine:
- Control-room displays
- Government forms
- Security camera overlays
- Technical diagnostics
- Municipal infrastructure maps
- Monospaced data terminals
- Minimal Swiss-style information hierarchy
- Machine-vision annotation systems

### Composition principles
1. **Grid as authority** — all layouts should feel governed by a rigid underlying structure.
2. **Data before atmosphere** — mood comes from information density and restraint, not decoration.
3. **Edges over softness** — use sharp geometry, hard boundaries, and precise containers.
4. **Status everywhere** — every object can have state, confidence, time, or classification.
5. **Surveillance perspective** — use overhead maps, cropped camera angles, target frames, and metadata overlays.

### Layout behavior
- Use dense but disciplined layouts.
- Prefer panels, columns, scanlines, overlays, and segmented control surfaces.
- Use vertical and horizontal dividers aggressively.
- Keep whitespace narrow and intentional.
- Use asymmetry only when it creates hierarchy or alarm.
- Avoid decorative empty space; empty space should feel like unused capacity.

## 4. Color System

### Palette philosophy
The color system should feel industrial, administrative, and sensor-driven. Most surfaces should be dark, gray, or desaturated. Accent colors should behave like system states rather than brand decoration.

### Core palette

| Token | Hex | Use |
|---|---:|---|
| `obsidian` | `#050608` | Primary background, command surfaces |
| `carbon` | `#101317` | Secondary background, cards, panels |
| `gunmetal` | `#1B2026` | Nested panels, inactive modules |
| `slate` | `#2A3139` | Borders, inactive states, dividers |
| `ash` | `#7D8792` | Secondary text, labels, captions |
| `whiteout` | `#E8EEF2` | Primary text on dark backgrounds |
| `blue-scan` | `#39C7F3` | Live system state, scan highlights, selected objects |
| `cyan-fog` | `#8EEBFF` | High-emphasis data, active trace lines |
| `acid-lime` | `#C8F23A` | Confirmed, complete, authorized |
| `amber-risk` | `#F5B74A` | Warning, degraded, pending review |
| `red-lock` | `#FF3B30` | Critical, denied, breached, hostile |
| `violet-signal` | `#8B5CF6` | Unknown, encrypted, synthetic, external signal |

### Color usage ratios
- 70% black / near-black system surfaces
- 20% gray structural UI
- 7% cyan active telemetry
- 2% amber/red risk states
- 1% lime or violet special-purpose indicators

### Accessibility
- Primary text on dark surfaces must meet WCAG AA contrast.
- State must never rely on color alone; pair with label, icon, code, or pattern.
- Use red sparingly so critical states retain force.
- Reserve bright cyan for interactive or live elements.

### Gradients
Use gradients only for:
- Radar sweeps
- Scanning effects
- Heat maps
- Depth in map overlays
- Signal intensity

Avoid decorative marketing gradients. If a gradient does not represent data, risk, light, or telemetry, remove it.

## 5. Typography

### Typography thesis
Typography should feel bureaucratic, technical, and machine-readable. It should not feel expressive. It should imply that the system privileges clarity, control, and auditability over warmth.

### Recommended type stack

#### Primary sans
Use a neutral grotesk or neo-grotesk:
- Helvetica Neue
- Arial Nova
- Inter Tight
- Univers-style substitutes
- Neue Haas Grotesk-style substitutes

Use for headlines, UI labels, navigation, and primary content.

#### Monospace
Use a technical monospace:
- IBM Plex Mono
- JetBrains Mono
- Roboto Mono
- Space Mono
- SF Mono

Use for IDs, timestamps, coordinates, telemetry, logs, system codes, and annotations.

### Type scale

| Token | Size | Line height | Use |
|---|---:|---:|---|
| `display-xl` | 72px | 0.9 | Hero statements, campaign impact |
| `display-lg` | 56px | 0.95 | Major section headers |
| `display-md` | 40px | 1.0 | Product headers |
| `heading-lg` | 28px | 1.1 | Panel group titles |
| `heading-md` | 20px | 1.2 | Module titles |
| `body-lg` | 18px | 1.45 | Marketing body |
| `body-md` | 15px | 1.45 | Default body |
| `body-sm` | 13px | 1.35 | Secondary body |
| `label` | 11px | 1.1 | Interface labels |
| `micro` | 9px | 1.0 | Metadata, system tags |

### Typographic rules
- Headlines should be short and declarative.
- Use uppercase for labels, statuses, coordinates, and classifications.
- Use monospaced numerals for all operational metrics.
- Keep letter spacing tight for large headlines and wider for labels.
- Avoid casual punctuation in interface copy.
- Avoid exclamation points entirely.

### Example typography pairing
- Hero: compressed neutral sans, 64px, uppercase optional.
- Subhead: neutral sans, 18px, regular, ash.
- Metadata: monospace, 11px, uppercase, cyan or ash.
- Alert label: monospace, 11px, uppercase, red or amber.

## 6. Logo and Mark Direction

### Logo concept
The Overlord mark should feel like a control aperture, authority seal, sensor node, or centralized command symbol. It should suggest hierarchy and observation without becoming a literal eye.

### Mark principles
- Geometric
- Symmetric or near-symmetric
- Built from lines, brackets, nodes, or nested frames
- Works as a small interface glyph
- Can be rendered as a stencil, LED display, or monochrome badge

### Logo usage
- Prefer monochrome white, black, or cyan.
- Use the mark as a system watermark in dashboards.
- Use low-opacity repetitions as security texture.
- Avoid friendly lockups or rounded wordmarks.

### Do not
- Use cartoonish eye icons.
- Use skulls, monsters, or villain imagery.
- Use military insignia too literally.
- Use decorative crests unless heavily abstracted.

## 7. Iconography

### Style
- Monoline or solid technical icons.
- 90-degree geometry where possible.
- Minimal curves.
- Square bounding boxes.
- Sharp terminals.
- Consistent 1.5px or 2px stroke.

### Icon categories
- Asset
- Identity
- Camera
- Node
- Route
- Perimeter
- Signal
- Alert
- Lock
- Key
- Sensor
- Map layer
- Intercept
- Audit
- Command
- Override

### Rules
- Icons should be functional, not illustrative.
- Pair icons with text in critical contexts.
- Icons can appear inside brackets or reticles.
- Avoid soft, hand-drawn, filled emoji-like icons.

## 8. Interface Components

### Buttons

#### Primary button
- Background: `blue-scan`
- Text: `obsidian`
- Shape: 2px radius or square
- Label: uppercase, monospace or tight sans
- Hover: brighter cyan outline with subtle scan animation
- Active: inset border, brief compression

Examples:
- AUTHORIZE
- DEPLOY NODE
- OPEN INCIDENT
- VERIFY SUBJECT

#### Secondary button
- Transparent background
- 1px border in `slate`
- Text: `whiteout`
- Hover: border shifts to `blue-scan`

#### Destructive button
- Background: transparent
- Border: `red-lock`
- Text: `red-lock`
- Hover: red fill at 12–18% opacity

Examples:
- REVOKE ACCESS
- QUARANTINE NODE
- LOCK SECTOR

### Cards and panels
Panels should feel like modular control surfaces.

#### Panel anatomy
- Header strip with title, ID, status, and timestamp
- Main content area for metrics, map, video, or log entries
- Footer row for actions or audit metadata
- Optional side rail for severity or classification

#### Panel rules
- Use 1px borders.
- Use hard shadows sparingly.
- Use nested panels for drill-down states.
- Do not over-round corners.
- Make inactive panels visually dim but still readable.

### Alerts

#### Alert levels
| Level | Color | Label | Behavior |
|---|---|---|---|
| Info | `blue-scan` | `SYSTEM` | Passive update |
| Success | `acid-lime` | `CONFIRMED` | Resolution or authorization |
| Warning | `amber-risk` | `DEGRADED` | Attention required |
| Critical | `red-lock` | `BREACH` | Immediate operator action |
| Unknown | `violet-signal` | `UNVERIFIED` | Requires classification |

#### Alert format
`[SEVERITY] [ASSET/LOCATION] [EVENT] [TIME] [ACTION]`

Examples:
- `BREACH / SECTOR 04 / ACCESS VIOLATION / 14:32:08 / LOCKDOWN AVAILABLE`
- `DEGRADED / NODE 17 / SIGNAL LOSS / 09:11:44 / ROUTE BACKUP`

### Tables
Tables are central to Overlord.

#### Table rules
- Dense rows are acceptable.
- Use sticky headers.
- Use monospace for values.
- Include filters, severity, status, and timestamp.
- Use row highlighting for active selection.
- Use right-aligned numerics.
- Provide export and audit actions.

### Forms
Forms should feel like controlled intake.

#### Rules
- Use labels above fields.
- Include required markers as system notation, not red asterisks when possible.
- Provide field-level validation with exact reason.
- Avoid conversational helper text.

Example:
- `AUTHORIZATION CODE REQUIRED`
- `FORMAT INVALID: EXPECTED 6-CHARACTER NODE ID`

## 10. Motion System

### Motion thesis
Motion should feel computational, not emotional. The system is scanning, locking, classifying, and escalating.

### Motion behaviors
- Scanline pass
- Reticle lock-on
- Panel slide-in from hard edges
- Data reveal through wipe or flicker
- Map pulse on event detection
- Alert escalation blink
- Progress as segmented loading
- Timestamp increments

### Timing
- Micro interactions: 80–120ms
- Panel transitions: 160–240ms
- Map scans: 800–1600ms
- Critical alerts: persistent pulse, not decorative bounce

### Easing
Use linear, stepped, or slightly mechanical easing. Avoid springy, playful, or elastic motion.

### Do not
- Use confetti.
- Use bouncy transitions.
- Use organic morphing.
- Animate purely for delight.

## 11. Imagery and Art Direction

### Image mood
Images should feel observational, institutional, and detached.

### Subject matter
- Security cameras
- Server rooms
- City infrastructure
- Traffic systems
- Industrial facilities
- Control centers
- Operators in silhouette
- Overhead urban grids
- Sensor arrays
- Access checkpoints
- Drones and autonomous assets
- Doors, gates, towers, antennas

### Treatment
- Crop tightly.
- Add metadata overlays.
- Use grayscale or desaturated color grading.
- Add subtle scanline texture.
- Use camera-frame UI elements.
- Avoid smiling subjects and staged collaboration scenes.

### Image rules
- People should appear as operators, subjects, or silhouettes.
- Do not use lifestyle photography.
- Do not over-humanize the system.
- Never use surveillance imagery casually; keep it purposeful and operational.

- Topographic gridlines
- Signal interference
- Security hatch patterns
- Map grid overlays
- Barcode-like strips
- Redaction blocks
- Low-opacity coordinate systems

### Pattern usage
Use texture as an atmospheric layer behind structured UI. It should never reduce legibility.

### Recommended pattern motifs
- Nested rectangles
- Bracket arrays
- Node grids
- Signal arcs
- Dotted tracking paths
- Microtext fields
- Repeating classification labels

## 13. Accessibility and Ethics

### Accessibility
- Preserve high contrast in all operational surfaces.
- Use labels and symbols in addition to color.
- Support reduced motion.
- Make alerts screen-reader compatible.
- Provide readable focus states.
- Avoid tiny text for primary actions.

### Ethical framing
Overlord can borrow the aesthetics of control, but the product should still communicate accountability.

Include:
- Audit trails
- Permission clarity
- Access logs
- Review states
- Human authorization checkpoints
- Data retention visibility
- Appeal or correction pathways where appropriate

The brand may feel powerful. It should not feel lawless.

## 14. UI Writing System

### Microcopy rules
- Use imperative verbs.
- Use exact nouns.
- Avoid humor.
- Avoid apologies unless legally or ethically necessary.
- Prefer status codes over emotional phrasing.

### Good examples
- `ACCESS DENIED: ROLE INSUFFICIENT`
- `NODE OFFLINE: LAST SIGNAL 08:42:17`
- `SUBJECT MATCH: 84% CONFIDENCE`
- `ESCALATION READY`
- `AUDIT LOG SEALED`

### Bad examples
- `Oops, something went wrong.`
- `Looks like we lost that node.`
- `Nice work! You’re all set.`
- `Let’s try that again.`

## 15. Web and Marketing Pages

### Homepage structure
1. Hero with severe outcome statement.
2. Live-system visual or command-dashboard mockup.
3. Three proof modules: visibility, classification, response.
4. Operational scenario walkthrough.
5. Security, compliance, and auditability section.
6. Technical architecture section.
7. Deployment model.
8. Final CTA.

### Page mood
- Cold and premium.
- Dense enough to feel real.
- Restrained enough to feel powerful.
- Avoid excessive narrative copy.

### CTA language
- REQUEST ACCESS
- VIEW SYSTEM
- START ASSESSMENT
- DEPLOY PILOT
- CONTACT OPERATIONS

## 16. Product Design Principles

### Principle 1: Show state before explanation
The operator should instantly know what is active, degraded, critical, unknown, or locked.

### Principle 2: Every object has metadata
Assets, people, locations, alerts, and actions should carry identifiers, timestamps, status, source, and confidence.

### Principle 3: Decision paths must be auditable
Every major action should leave a trace.

### Principle 4: Reduce ambiguity under pressure
UI should become more direct as severity increases.

### Principle 5: Authority requires accountability
Powerful workflows must include permissions, confirmations, logs, and review.

## 18. Example Component Spec

**Purpose:** summarize one live event requiring operator review.

**Required elements:**
- Severity label
- Incident ID
- Location or asset
- Timestamp
- Confidence score
- Source count
- Current status
- Primary action
- Audit link

**Visual style:**
- Dark panel
- 1px slate border
- Red or amber severity rail
- Monospace metadata
- Cyan selected state

**Sample content:**

## 19. Brand Checks

Before shipping any Overlord asset, ask:
- Does this feel controlled rather than excited?
- Is every important object clearly classified?
- Does the layout imply authority and system awareness?
- Are alerts unmistakable without being theatrical?
- Is the motion functional rather than decorative?
- Does the system preserve accountability?
- Would this still work in a command center at 3:00 AM?

## 20. One-Line Summary

Overlord is a severe command design system for organizations that need to turn fragmented signals into controlled, auditable action.
